# Stock V1 Refactor Rollback Runbook

Last updated: 2026-04-09

## Purpose

Provide a fast rollback path for stock read behavior regressions while keeping additive schema migrations in place.

## Safety Baseline

1. Keep the latest DB backup/snapshot artifact generated by:
   - `npm run db:snapshot`
2. Do not remove additive migrations (`030` to `034`) during app rollback.
3. Roll back application behavior first, database rollback only for critical corruption.

## Fast Rollback Paths

### 1) Revert to previous application build

Use your deployment platform rollback command to move to the last stable app artifact while leaving schema intact.

### 2) Revert prod-only read default behind feature flag

Set:
- `STOCKS_V1_DEFAULT_DATASET_SCOPE=all`

Behavior:
- `prod` (default): stock read paths apply `dataset_type='prod'`.
- `all`: stock read paths do not enforce a default dataset filter unless request explicitly sets `dataset=prod|test`.

After updating environment variables, restart backend processes.

## Validation After Rollback

Run:

```bash
npm run db:check
npm run stocks:v1:verify:dataset-scope
npm run stocks:v1:regression
npm run stocks:v1:monitor -- --minProd1dRows24h=0 --maxProbeLatencyMs=2000
```

## Critical Corruption Path (DB Restore)

Only if data integrity is compromised:

1. Stop write traffic.
2. Restore from latest snapshot/backup artifact produced by `npm run db:snapshot`.
3. Re-run `npm run db:check` and `npm run stocks:v1:validate` before reopening traffic.
